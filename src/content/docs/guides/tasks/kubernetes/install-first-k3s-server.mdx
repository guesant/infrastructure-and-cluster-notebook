> **Pré-requisitos:** host preparado conforme [preparar um servidor Debian](../../host/prepare-debian-server/), acesso root.
> **Versões testadas:** K3s v1.36.1+k3s1, Debian 12 (bookworm).

Esta etapa produz um cluster funcional de nó único. Em uma topologia multinó, ela também inicializa o primeiro manager e o datastore etcd ao qual os demais servidores e agentes serão conectados.

## Instalação com arquivo de configuração persistente

Para o primeiro servidor, o roteiro recomendado por este guia grava as opções em
`/etc/rancher/k3s/config.yaml` em vez de passá-las só na linha de comando: isso facilita
reinstalações, auditoria e integração com backups do host.

Execute em um host novo. Por padrão, cada valor é perguntado no terminal via `read`; o token do
cluster já vem configurado para ser gerado ao executar (`openssl rand`), sem aparecer nesta
página. Cada campo pode ter o modo trocado no menu "Personalizar" (embutido no script gerado,
criptografado com `age`, ou perguntado como os demais) e o corpo do script pode ser editado antes
de copiar.

## Validação

> **Executar em:** o mesmo host, como `root`.

```bash
systemctl status k3s
k3s kubectl get nodes
k3s kubectl get pods --all-namespaces
```

O serviço deve estar `active (running)`, o nó deve aparecer como `Ready` e os Pods de sistema (`coredns`, `traefik`, `metrics-server` quando não desabilitados) devem estar `Running` ou `Completed`.

## Troubleshooting

Se o nó não aparecer como `Ready`, verifique `journalctl -u k3s -n 100`: a causa mais comum é um `config.yaml` inválido (indentação YAML) ou uma porta já ocupada. Se os Pods de `kube-system` ficarem presos em `Pending`, confirme que os requisitos de [validar requisitos do host](../../host/validate-host-requirements/) foram atendidos, em especial cgroups e módulos de kernel.
