> **Pré-requisitos:** kubeconfig com acesso de escrita ao namespace alvo.
> **Versões testadas:** Kubernetes 1.36.

Um Secret criado diretamente via `kubectl` é a forma mais simples de fornecer uma credencial a um Pod, mas não é versionado: é a escolha certa para segredos de bootstrap ou de teste, nunca para o estado permanente de uma aplicação gerenciada por GitOps. Para o caminho versionado, veja [SOPS com age](../configure-sops-with-age/) ou [Sealed Secrets](../install-sealed-secrets/).

## Criar a partir de literais

> **Executar em:** qualquer máquina com `KUBECONFIG` e acesso de escrita ao namespace.

```bash
read -r -p "Namespace: " SECRET_NAMESPACE
read -r -p "Nome do Secret: " SECRET_NAME
read -r -s -p "Valor da credencial: " SECRET_VALUE
printf '\n'

kubectl --namespace "${SECRET_NAMESPACE}" create secret generic "${SECRET_NAME}" \
  --from-literal=value="${SECRET_VALUE}"
```

## Criar a partir de um arquivo

```bash
read -r -p "Caminho do arquivo com o valor: " SECRET_FILE_PATH

kubectl --namespace "${SECRET_NAMESPACE}" create secret generic "${SECRET_NAME}" \
  --from-file="${SECRET_FILE_PATH}"
```

## Validação

> **Executar em:** qualquer máquina com `KUBECONFIG` e acesso de leitura ao namespace.

```bash
kubectl --namespace "${SECRET_NAMESPACE}" get secret "${SECRET_NAME}"
```

Não use `kubectl get secret -o yaml` para "conferir o valor" em uma tela compartilhada: isso imprime o valor codificado em base64, trivialmente decodificável.

## Troubleshooting

Se um Pod não conseguir ler o Secret montado, confirme o namespace (Secrets não são compartilhados entre namespaces) e a permissão RBAC da ServiceAccount usada pelo Pod.

## Rollback

```bash
kubectl --namespace "${SECRET_NAMESPACE}" delete secret "${SECRET_NAME}"
```
