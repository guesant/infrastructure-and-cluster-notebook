> **Pré-requisitos:** [OpenBao já instalado](../install-openbao/), acesso a um KMS externo (AWS, GCP ou Azure).
> **Versões testadas:** OpenBao 2.4.

O [unseal manual](../install-openbao/#fazer-o-unseal) exige um operador presente com as chaves Shamir sempre que o Pod do OpenBao reinicia; isso não escala em produção e cria um ponto único de indisponibilidade se o operador não estiver disponível. Auto-unseal delega essa etapa a um serviço KMS externo: o OpenBao passa a se desencriptar sozinho na inicialização, usando uma chave que o KMS gerencia fora do cluster. Este guia cobre o setup com **AWS KMS**; os blocos de configuração para GCP e Azure aparecem como alternativas ao final da etapa de configuração.

## Criar a IAM role e as credenciais

OpenBao precisa de permissões para acessar a chave KMS. Crie uma role dedicada:

<ScriptHelper
  runWhere="máquina com `aws` CLI configurado"
  script={openbaoAwsIamSetupScript}
  fields={[
    { var: 'KMS_KEY_ARN', label: 'ARN da chave KMS (da etapa anterior)', defaultValue: 'arn:aws:kms:us-east-1:123456789:key/abc-def-ghi' },
    { var: 'IAM_ROLE_NAME', label: 'Nome da role IAM', defaultValue: 'openbao-unseal' },
  ]}
/>

Se o OpenBao rodar em Kubernetes (K3s), use IRSA (IAM Roles for Service Accounts) para associar essa role à ServiceAccount do Pod, em vez de embutir credenciais estáticas de longa duração no container.

## Atualizar a configuração do OpenBao

Edite o arquivo de configuração do OpenBao (`/etc/openbao/openbao.hcl`, ou os values do Helm se a instalação for via chart):

<FileWriter path="openbao.hcl" lang="hcl">
{`ui = true

storage "postgresql" {
  connection_string = "postgres://bao:password@postgres-host:5432/openbao"
}

seal "awskms" {
  region     = "us-east-1"
  kms_key_id = "arn:aws:kms:us-east-1:123456789:key/abc-def-ghi"
}

listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/etc/openbao/certs/bao.crt"
  tls_key_file  = "/etc/openbao/certs/bao.key"
}`}
</FileWriter>

**Para GCP (Google Cloud KMS):**

<FileWriter path="openbao-gcp.hcl" lang="hcl">
{`seal "gcpkms" {
  project = "my-project"
  region  = "us-central1"
  key_ring = "openbao-ring"
  crypto_key = "openbao-key"
}`}
</FileWriter>

**Para Azure:**

<FileWriter path="openbao-azure.hcl" lang="hcl">
{`seal "azurekeyvault" {
  tenant_id      = "your-tenant-id"
  client_id      = "your-client-id"
  client_secret  = "your-client-secret"
  vault_name     = "my-vault"
  key_name       = "openbao-unseal-key"
}`}
</FileWriter>

## Aplicar a configuração

> **Executar em:** o host ou Pod onde o OpenBao roda.

```bash
sudo systemctl restart openbao
# ou, se instalado via Helm no cluster:
kubectl --namespace openbao rollout restart deployment/openbao
```

Aguarde a inicialização e confirme que o OpenBao desencriptou sem intervenção manual:

```bash
bao status
```

A saída deve mostrar `Seal Type: awskms` e `Sealed: false` sem que nenhum comando de unseal tenha sido executado nesta sessão; esse é o efeito observável do auto-unseal.

## Validação

Teste o cenário que o auto-unseal existe para resolver: um reinício não planejado sem operador disponível para desbloquear manualmente.

> **Executar em:** o host ou Pod onde o OpenBao roda.

```bash
sudo systemctl stop openbao
sleep 5
sudo systemctl start openbao
bao status
```

`Sealed` deve voltar a `false` sozinho, sem que nenhuma chave de unseal tenha sido informada durante o teste.

## Troubleshooting

Comece verificando o `Seal Type` reportado por `bao status` antes de investigar a rede: se ele mostrar `shamir` em vez de `awskms`, a configuração com o bloco `seal` não foi recarregada; confirme que o arquivo aplicado corresponde ao mostrado acima (`cat /etc/openbao/openbao.hcl`) e reinicie o serviço completamente (parar e só depois iniciar, não `restart`).

Se o erro for de permissão ao conectar no KMS, confirme que a role IAM tem a política de uso da chave e que a instância ou o Pod realmente assume essa role:

```bash
aws iam get-role-policy --role-name openbao-unseal --policy-name allow-kms-unseal
kubectl --namespace openbao describe pod openbao-0 | grep -i iam
```

Se o erro indicar que a chave KMS não foi encontrada, confirme que o ARN configurado ainda existe e não foi excluído:

```bash
aws kms describe-key --key-id "${KMS_KEY_ARN}" --region us-east-1
```
