> **Pré-requisitos:** kubeconfig com acesso administrativo à API.
> **Versões testadas:** OpenBao 2.4.

Veja [OpenBao e Vault](../../../../learn/secrets-management/openbao-and-vault/) antes de instalar, em especial a discussão sobre unseal e por que rodar o OpenBao dentro do próprio cluster que ele protege exige um plano explícito de recuperação.

## Instalar via Helm

> **Executar em:** qualquer máquina com `KUBECONFIG`, Helm e acesso administrativo à API.

```bash
read -r -p "Versão do chart openbao: " OPENBAO_VERSION

helm upgrade --install openbao openbao \
  --repo https://openbao.github.io/openbao-helm \
  --version "${OPENBAO_VERSION}" \
  --namespace openbao \
  --create-namespace \
  --set server.dev.enabled=false \
  --set server.standalone.enabled=true
```

`server.dev.enabled=false` é deliberado: o modo dev nunca persiste dados e nunca deve ser usado além de testes descartáveis.

## Inicializar e obter as chaves de unseal

> **Executar em:** qualquer máquina com `KUBECONFIG` e acesso ao Pod do OpenBao.

```bash
kubectl --namespace openbao exec -it openbao-0 -- bao operator init
```

A saída contém as chaves de unseal e o token root inicial: **capture-as imediatamente** e distribua-as conforme a política de custódia decidida (veja [estratégias de recuperação](../../../../learn/secrets-management/recovery-strategies/)). Elas não são exibidas novamente.

## Fazer o unseal

```bash
read -r -p "Uma das chaves de unseal: " UNSEAL_KEY

kubectl --namespace openbao exec -it openbao-0 -- bao operator unseal "${UNSEAL_KEY}"
```

Repita com chaves adicionais até atingir o limiar mínimo configurado (padrão 3 de 5).

## Habilitar o secrets engine KV

> **Executar em:** com o CLI `bao` autenticado (token root ou identidade equivalente).

```bash
bao secrets enable -path=secret kv-v2
```

## Validação

```bash
kubectl --namespace openbao exec -it openbao-0 -- bao status
```

Deve mostrar `Sealed: false` e o número de chaves fornecidas atingindo o limiar configurado.

## Troubleshooting

Se o Pod reiniciar, o OpenBao volta ao estado `sealed`: refaça o unseal manualmente ou configure auto-unseal via KMS de nuvem para eliminar essa etapa manual recorrente.

## Rollback

```bash
helm --namespace openbao uninstall openbao
```

:::danger
Desinstalar remove o Pod, mas os dados persistidos em volume podem continuar existindo dependendo da `reclaimPolicy` da StorageClass; confirme antes de considerar os dados removidos.
:::
