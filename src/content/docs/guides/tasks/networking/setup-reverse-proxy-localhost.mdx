> **Pré-requisitos:** [CoreDNS configurado para a zona `internal`](../setup-coredns-internal/), acesso root ao host que vai rodar o proxy (bastion ou laptop administrativo), Nginx ou HAProxy instalado.
> **Versões testadas:** Nginx 1.25, HAProxy 2.8.

O guia anterior faz `grafana.internal` resolver para `127.0.0.1`. Essa resolução sozinha não abre nenhuma conexão: falta um processo escutando em `127.0.0.1:443` que receba a conexão TLS e decida, a partir do nome pedido pelo cliente, para qual serviço interno encaminhar. É esse processo que este guia configura, usando o campo SNI (Server Name Indication, o nome de domínio que o cliente TLS envia antes da negociação, visível mesmo sem descriptografar a conexão) para escolher o destino.

Repare que os nomes administrativos (`grafana.internal`) e os nomes reais dos Services no cluster (`grafana-svc.default.svc.cluster.local`) pertencem a espaços de nomes diferentes: o primeiro é a convenção local criada no guia anterior para o operador digitar; o segundo é a resolução real de Service do Kubernetes, usada pelo proxy para alcançar o backend. As duas opções abaixo resolvem o mesmo problema com desenhos diferentes: a Opção A (Nginx) termina o TLS no proxy usando um certificado autoassinado compartilhado; a Opção B (HAProxy) inspeciona o SNI sem descriptografar e repassa a conexão TLS intacta até o backend. Escolha a Opção A quando quiser um único certificado simples para todo o acesso administrativo; escolha a Opção B quando os backends já tiverem certificados próprios válidos e o objetivo for preservá-los ponta a ponta.

## Opção A: Nginx (termina TLS no proxy)

Gere um certificado autoassinado cobrindo os nomes administrativos que serão publicados:

> **Executar em:** host que vai rodar o proxy, como `root`.

```bash
mkdir -p /etc/nginx/certs
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/nginx/certs/internal.key \
  -out /etc/nginx/certs/internal.crt \
  -subj "/CN=*.internal" \
  -addext "subjectAltName=DNS:grafana.internal,DNS:prometheus.internal,DNS:argocd.internal"
```

`subjectAltName` precisa listar cada nome administrativo explicitamente (um `CN` com curinga sozinho não é suficiente para navegadores modernos); acrescente um `DNS:` por serviço adicionado no guia anterior e regenere o certificado quando a lista mudar.

Crie o arquivo de configuração do site:

```nginx
# /etc/nginx/sites-available/internal-proxy
upstream grafana_backend {
    server grafana-svc.default.svc.cluster.local:3000;
}

upstream prometheus_backend {
    server prometheus-svc.default.svc.cluster.local:9090;
}

upstream argocd_backend {
    server argocd-server.argocd.svc.cluster.local:443;
}

server {
    listen 127.0.0.1:443 ssl;
    server_name grafana.internal prometheus.internal argocd.internal;

    ssl_certificate     /etc/nginx/certs/internal.crt;
    ssl_certificate_key /etc/nginx/certs/internal.key;

    # Os backends usam certificados internos do cluster; desabilitar a
    # verificação evita falhas por CA não reconhecida nesta camada.
    proxy_ssl_verify off;

    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    location / {
        if ($host = grafana.internal) {
            proxy_pass http://grafana_backend;
        }
        if ($host = prometheus.internal) {
            proxy_pass http://prometheus_backend;
        }
        if ($host = argocd.internal) {
            proxy_pass https://argocd_backend;
        }
    }
}
```

Ative o site e valide a sintaxe antes de recarregar:

> **Executar em:** host que vai rodar o proxy, como `root`.

```bash
ln -sf /etc/nginx/sites-available/internal-proxy /etc/nginx/sites-enabled/internal-proxy
nginx -t
systemctl reload nginx
```

`nginx -t` detecta erros de sintaxe antes de qualquer interrupção; nunca recarregue sem essa checagem passar primeiro.

## Opção B: HAProxy (passthrough por SNI)

Diferente do Nginx acima, o HAProxy aqui não termina TLS: ele apenas inspeciona o SNI da negociação e encaminha os bytes originais, criptografados, até o backend. Não é necessário nenhum certificado no HAProxy.

Configure o frontend e os backends:

> **Executar em:** host que vai rodar o proxy, como `root`.

```text
# /etc/haproxy/haproxy.cfg
global
    log stdout local0

defaults
    mode tcp
    timeout connect 5s
    timeout client 50s
    timeout server 50s

frontend internal_https
    bind 127.0.0.1:443
    tcp-request inspect-delay 2s
    tcp-request content accept if { req.ssl_hello_type 1 }

    use_backend grafana_backend if { req.ssl_sni -i grafana.internal }
    use_backend prometheus_backend if { req.ssl_sni -i prometheus.internal }
    use_backend argocd_backend if { req.ssl_sni -i argocd.internal }
    default_backend grafana_backend

backend grafana_backend
    server grafana grafana-svc.default.svc.cluster.local:3000 check

backend prometheus_backend
    server prometheus prometheus-svc.default.svc.cluster.local:9090 check

backend argocd_backend
    server argocd argocd-server.argocd.svc.cluster.local:443 check
```

Como o tráfego chega intacto ao backend, cada serviço precisa apresentar seu próprio certificado válido para essa conexão (o Argo CD, por exemplo, atende TLS diretamente na porta 443 do Service `argocd-server`); não há um certificado único do proxy cobrindo todos os nomes, ao contrário da Opção A.

Valide a configuração e recarregue:

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
systemctl reload haproxy
```

## Validação

> **Executar em:** host que roda o proxy, ou qualquer estação que resolva os nomes administrativos pela zona configurada no guia anterior.

```bash
curl -k https://grafana.internal
```

Deve retornar a página de login do Grafana (ou o corpo esperado do serviço testado). `-k` ignora o aviso de certificado autoassinado da Opção A; na Opção B (passthrough), o certificado apresentado é o do backend, então `-k` só é necessário se esse certificado também não for confiável pela estação cliente.

## Troubleshooting

Comece verificando se o proxy está de fato escutando antes de suspeitar de roteamento:

```bash
ss -tlnp 'sport = :443'
```

Se nada aparecer, o serviço (`nginx` ou `haproxy`) não subiu; revise `journalctl -u nginx` ou `journalctl -u haproxy` para o erro de inicialização. Se o proxy está escutendo mas a conexão retorna erro de destino errado ou 404, confirme que o `server_name`/`req.ssl_sni` corresponde exatamente ao nome usado no `curl`, incluindo a ausência de `www.` ou de subdomínios extras. Se a conexão chega ao proxy mas falha ao alcançar o backend, teste a resolução do nome de Service Kubernetes diretamente a partir do host do proxy; se ele não estiver na rede de Pods do cluster, esses nomes internos do Kubernetes não são alcançáveis dali, e o proxy precisa rodar dentro do cluster ou ter uma rota específica até a rede de Pods.

## Rollback

```bash
rm -f /etc/nginx/sites-enabled/internal-proxy
systemctl reload nginx
```

Para HAProxy, restaure a configuração anterior a partir de um backup ou remova o `frontend internal_https` e os backends associados antes de recarregar.
