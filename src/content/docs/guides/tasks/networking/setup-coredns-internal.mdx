> **Pré-requisitos:** kubeconfig administrativo, CoreDNS disponível no cluster (padrão em qualquer instalação K3s).
> **Versões testadas:** K3s v1.36.1+k3s1, CoreDNS 1.11 (empacotado pelo K3s).

O CoreDNS do K3s já resolve os nomes reais do cluster (Services e Pods) sob a zona `cluster.local`, através do plugin `kubernetes` configurado no Corefile padrão. Esta página adiciona uma **zona separada**, não relacionada a `cluster.local`, para nomes administrativos como `grafana.internal`, que o [reverse proxy local](../setup-reverse-proxy-localhost/) resolve depois para rotear tráfego de operadores até os serviços internos.

Não reaproveite `cluster.local` para esses nomes administrativos. Declarar uma zona adicional exatamente para `cluster.local` no mesmo Corefile entra em conflito com o plugin `kubernetes` que já é autoritativo para essa zona; dependendo de como o bloco é declarado, a nova zona pode ter prioridade sobre o plugin `kubernetes` e quebrar a resolução real de Services e Pods do cluster inteiro. Este guia usa o sufixo `.internal`, reservado pela IANA especificamente para nomes internos de organizações ([RFC 9476](https://www.rfc-editor.org/rfc/rfc9476)), o que evita tanto essa colisão quanto o sufixo `.local`, reservado para mDNS ([RFC 6762](https://www.rfc-editor.org/rfc/rfc6762)) e que pode ser interceptado por Avahi ou Bonjour em algumas estações administrativas.

## Confirmar o CoreDNS

> **Executar em:** estação administrativa com `KUBECONFIG` e acesso à API.

```bash
kubectl --namespace kube-system get deployment coredns
kubectl --namespace kube-system get service coredns
```

O Deployment deve estar `1/1` pronto. Anote o `CLUSTER-IP` do Service `coredns`: ele é o endereço que uma estação fora do cluster precisa alcançar para consultar essa zona, o que exige rede roteável até o cluster (VPN, rede do bastion) ou expor o Service por `NodePort`, conforme discutido na validação abaixo.

## Adicionar a zona `internal`

O CoreDNS lê sua configuração do `ConfigMap` `coredns`, no campo `Corefile`. Uma zona nova precisa ser um **bloco de servidor irmão** do bloco `.:53` existente, nunca aninhado dentro dele: o Corefile não aceita blocos de zona aninhados, e um `local:53 { ... }` escrito dentro das chaves do bloco `.:53` é um erro de sintaxe que impede o CoreDNS de iniciar.

Para um punhado de nomes administrativos, o plugin [`hosts`](https://coredns.io/plugins/hosts/) resolve diretamente a partir de entradas escritas no próprio Corefile, sem exigir um arquivo de zona externo montado no Pod (a abordagem com o plugin `file`, que lê um arquivo separado, exige montar esse arquivo como volume no Deployment do CoreDNS, o que a maioria dos ambientes K3s não tem preparado por padrão).

Reaplique o `ConfigMap` preservando o bloco `.:53` original e acrescentando o bloco `internal:53` ao final:

> **Executar em:** estação administrativa com `KUBECONFIG` e acesso administrativo à API.

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
        }
        prometheus :9153
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
    internal:53 {
        errors
        hosts {
            127.0.0.1 grafana.internal
            127.0.0.1 prometheus.internal
            127.0.0.1 argocd.internal
            fallthrough
        }
    }
EOF
```

O primeiro bloco é o padrão de fábrica do K3s; confirme que ele corresponde ao Corefile atual do cluster antes de sobrescrever (`kubectl --namespace kube-system get configmap coredns -o jsonpath='{.data.Corefile}'`), porque instalações com componentes desabilitados ou versões diferentes de K3s podem ter variações. O segundo bloco é a zona nova: cada linha do `hosts` mapeia um nome ao IP do reverse proxy que vai atendê-lo, que é `127.0.0.1` na topologia recomendada (o proxy roda no mesmo bastion ou laptop que faz a consulta DNS). Acrescente uma linha por serviço interno que o [reverse proxy](../setup-reverse-proxy-localhost/) publicar.

Recarregue o CoreDNS para aplicar a mudança:

```bash
kubectl --namespace kube-system rollout restart deployment coredns
kubectl --namespace kube-system rollout status deployment coredns --timeout=120s
```

## Validação

Teste a resolução de dentro do cluster primeiro, isolando problemas de rede externa:

> **Executar em:** estação administrativa com `KUBECONFIG` e acesso à API.

```bash
kubectl run dns-test --rm -it --restart=Never --image=busybox:1.36 -- nslookup grafana.internal
```

Deve retornar `127.0.0.1`. Se esse teste falhar, o problema está no Corefile ou no CoreDNS, antes mesmo de considerar o acesso externo.

Para consultar a partir de uma estação fora do cluster (o cenário real de uso), aponte a consulta diretamente para o `CLUSTER-IP` anotado antes:

```bash
dig @<CLUSTER-IP-do-coredns> grafana.internal
```

Se essa consulta expirar sem resposta, o `CLUSTER-IP` provavelmente não é roteável a partir da estação administrativa: ClusterIPs geralmente só são alcançáveis de dentro da rede de Pods, a menos que o CNI use roteamento nativo ou a estação seja ela mesma um nó do cluster. Nesse caso, exponha o CoreDNS por um `Service` do tipo `NodePort` (`kubectl --namespace kube-system patch service coredns --type=merge --patch '{"spec":{"type":"NodePort"}}'`) e consulte pelo IP de um nó na porta atribuída, em vez do `CLUSTER-IP`.

## Troubleshooting

Comece pela verificação mais barata antes de suspeitar de rede: os logs do próprio CoreDNS.

```bash
kubectl --namespace kube-system logs deployment/coredns --tail=50
```

Um erro de parsing do Corefile aparece aqui imediatamente após o `rollout restart`, geralmente com a linha exata do problema; a causa mais comum é reintroduzir um bloco aninhado ou uma chave incorreta ao editar manualmente em vez de reaplicar o manifesto acima. Se os logs estiverem limpos mas a consulta ainda retornar `NXDOMAIN`, confirme que o nome consultado está exatamente entre as entradas do bloco `hosts` (o plugin não faz correspondência parcial) e que o `rollout restart` realmente terminou antes do teste.

## Rollback

Reaplique o Corefile original, com apenas o bloco `.:53` de fábrica:

```bash
kubectl apply -f - <<'EOF'
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
           lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
           pods insecure
           fallthrough in-addr.arpa ip6.arpa
        }
        prometheus :9153
        forward . /etc/resolv.conf {
           max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
EOF
kubectl --namespace kube-system rollout restart deployment coredns
```
