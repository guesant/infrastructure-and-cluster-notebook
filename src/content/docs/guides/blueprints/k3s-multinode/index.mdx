> **Público-alvo:** quem tem experiência com o [blueprint single-node](../k3s-single-node-gitops/) e quer escalar para múltiplos nós com alta disponibilidade.
> **Versões testadas:** K3s 1.36, Debian 12, topologia 3 servidores + N agentes.

Este blueprint estende o caminho single-node (Fase 2) para clusters K3s distribuídos em múltiplos
nós. Os conceitos fundamentais (Gateway API, cert-manager, Argo CD, Longhorn) permanecem os
mesmos; o que muda é a topologia do control plane, a coordenação entre nós via consenso Raft e os
runbooks de operação e recuperação.

- [Topologias recomendadas](topologies/): escolher entre 1 servidor + agentes, 3 servidores HA ou datastore externo.
- [Arquitetura de quorum](architecture/): como o etcd coordena servidores, o que é um quorum e os riscos de falha.
- [Requisitos de rede e portas](network-requirements/): comunicação entre nós, balanceamento da API, certificados TLS multi-SAN.
- [Inicializar e adicionar nós](first-server/): bootstrap do primeiro servidor e registro de servidores/agentes adicionais.
- [Balancear a API](api-endpoint/): endpoint estável para acesso administrativo e cliente.
- [Validar o cluster](validation/): checklist pós-implantação.
- [Operação nó a nó](node-maintenance/): manutenção e atualização sem downtime.
- [Falha e recuperação](failure-and-recovery/): cenários de perda de servidor, perda de agente e perda de quorum.

## Reaproveitamento da Fase 2

A maioria das páginas de task guides da Fase 2 é reutilizada (host, firewall, K3s, rede,
certificados). Este blueprint aponta para elas e descreve apenas as adaptações específicas de
multinode:

- [Adicionar um servidor K3s](../../tasks/kubernetes/join-k3s-server/): a primeira parte da instalação é igual à do single-node; diverge a partir do momento em que o novo servidor precisa se juntar ao quorum existente.
- [Configurar firewall entre nós](../../tasks/kubernetes/configure-k3s-firewall-rules/): as regras específicas de comunicação entre servidores estão detalhadas em [Requisitos de rede e portas](network-requirements/).
- [Configurar cert-manager com SANs](../../tasks/certificates/install-cert-manager/): necessário para que o certificado da API cubra múltiplos nomes, não apenas `localhost`.

## Diferenças do single-node

| Aspecto | Single-node | Multinode |
| --- | --- | --- |
| Control plane | 1 servidor | 3+ servidores (recomendado) ou 1 com datastore externo |
| Quorum | N/A | Número ímpar de servidores; perda de > 50% dos servidores = cluster travado |
| API endpoint | localhost ou IP do nó único | Load balancer ou DNS com múltiplos IPs |
| Workloads | Rodam sempre no nó único (sem redundância) | Distribuídos entre agentes; servidores podem ter workloads |
| Armazenamento | PVs ficam locais ao nó (sem replicação automática) | PVs replicados via Longhorn entre nós (se configurado) |
| Atualização | Reboot único | Rolling update: drain + update + uncordon nó a nó |
| Recuperação | Perda do nó = perda total | Perda de 1 agente = workloads rescheduled; perda de 1 servidor = cluster funciona se quorum > 50% |
