> **Para quem é:** quem está instalando K3s multinode e precisa inicializar o controle plane.
> **Pré-requisitos:** [topologias recomendadas](../topologies/), [requisitos de rede](../network-requirements/), hosts preparados com Debian/Ubuntu + UFW.

O primeiro servidor é o bootstrap do control plane: ele inicia o etcd, cria os tokens de join para novos nós e fica responsável pelo estado inicial do cluster. Este guia cobre:

1. Instalar o primeiro servidor (servidor-0)
2. Recuperar o token de join para adicionar servidores e agentes
3. Validação rápida (API acessível?)

## Passo 1: Instalar o primeiro servidor

Escolha sua topologia (3 servidores com etcd embarcado? ou com datastore externo?) e siga [Instalar o primeiro servidor K3s](../../../tasks/kubernetes/install-first-k3s-server/), com a adaptação:

**Se está usando 3 servidores com etcd embarcado**, adicione ao comando de instalação:

```bash
curl -sfL https://get.k3s.io | sh -s - server \
  --cluster-init \
  --write-kubeconfig-mode 644
```

A flag `--cluster-init` ativa o etcd embarcado e prepara este servidor para receber outros
servidores via join.

**Se está usando datastore externo (Postgres ou etcd)**, o comando é:

```bash
curl -sfL https://get.k3s.io | sh -s - server \
  --datastore-endpoint='postgres://user:password@postgres-host:5432/k3s' \
  --write-kubeconfig-mode 644
```

(Substitua o endpoint pela sua URL de datastore; veja [K3s: External Datastore](https://docs.k3s.io/datastore/external-datastore) para detalhes.)

## Passo 2: Recuperar o token de join

Depois que o primeiro servidor inicia, obtenha o token de join para adicionar novos
servidores/agentes:

```bash
sudo cat /var/lib/rancher/k3s/server/node-token
```

Saída esperada:

```text
K102abcdef12345...::server:zyx9876543210...
```

Guarde esse token: ele é necessário para adicionar servidores e agentes, e o comando acima é a
forma de recuperá-lo depois, caso precise novamente.

## Passo 3: Verificar que a API está rodando

Aguarde de 10 a 15 segundos enquanto o K3s inicia:

```bash
sudo kubectl get nodes
```

Saída esperada (apenas servidor-0):

```text
NAME         STATUS   ROLES                  AGE   VERSION
servidor-0   Ready    control-plane,master   30s   v1.36.0+k3s1
```

## Passo 4: Adicionar servidores e agentes

Agora que o primeiro servidor está pronto, siga:

- [Adicionar um servidor K3s](../../../tasks/kubernetes/join-k3s-server/): para servidor-1, servidor-2 etc.
- [Adicionar um agente K3s](../../../tasks/kubernetes/join-k3s-agent/): para workers.

Ambos os guias usam o token que você recuperou no Passo 2 e apontam para o primeiro servidor como `server-url`.

## Quorum e consenso

O quorum evolui conforme servidores são adicionados: com o 1º servidor, ainda não existe quorum
formado (o etcd ainda não tem outros membros para sincronizar). Com o 2º, o quorum passa a exigir
2 membros ativos, o que significa que nenhum dos dois pode falhar isoladamente sem travar o
cluster; é um estado transitório indesejável para permanecer por muito tempo. Só com o 3º servidor
o cluster atinge a configuração-alvo deste blueprint: quorum = 2, o que permite perder qualquer 1
dos 3 servidores sem perder a capacidade de tomar decisões (ver
[Arquitetura de quorum e etcd](../architecture/) para o raciocínio completo).

A forma mais confiável de confirmar que o quorum está saudável é verificar que todos os
servidores aparecem como `Ready`:

```bash
sudo kubectl get nodes
```

Inspecionar o etcd diretamente (listar membros do Raft, por exemplo) é possível, mas o comando
exato varia entre versões do K3s; consulte a
[documentação de HA com etcd embarcado](https://docs.k3s.io/datastore/ha-embedded) para a sintaxe
da versão instalada, em vez de depender de um comando fixo aqui.
