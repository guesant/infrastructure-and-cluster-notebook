As ferramentas desta seção são clientes administrativos. Elas podem ser instaladas em um manager ou em uma estação externa; pertencer ao cluster não é requisito. Para operar recursos remotamente, a máquina precisa alcançar a API correspondente e possuir credenciais com as permissões necessárias.

Os scripts deste repositório instalam binários em `/usr/local/bin` e usam `sudo` quando necessário. Revise o conteúdo antes de executar um script remoto. Para maior controle, clone o repositório e execute o arquivo localmente.

## kubectl

`kubectl` é o cliente principal da API Kubernetes. Ele cria, consulta, altera e remove objetos, acompanha logs e eventos e auxilia no diagnóstico de workloads. O comando lê o kubeconfig para decidir com qual cluster e identidade trabalhar; portanto, confira o contexto antes de qualquer alteração. O `kubectl` não acessa diretamente o banco do K3s nem os containers: suas operações passam pela API e pelas regras de autorização do Kubernetes. Referência: [introdução ao kubectl](https://kubernetes.io/docs/reference/kubectl/introduction/).

**Modelo de acesso e privilégios:** toda operação passa pelo RBAC do Kubernetes associado à identidade ativa no kubeconfig; `kubectl` não tem privilégio próprio além do que essa identidade já tem na API. **Riscos:** um kubeconfig com contexto de produção selecionado por engano é a causa mais comum de um comando destrutivo aplicado no cluster errado; confira `kubectl config current-context` antes de qualquer alteração. **Licença e plataformas:** Apache 2.0. Linux, macOS e Windows.

A instalação oficial baixa o binário publicado em `dl.k8s.io` e valida seu checksum antes de instalá-lo, em vez de confiar apenas no download:

> **Executar em:** máquina onde a CLI será instalada.

```bash
KUBECTL_VERSION="$(curl -Ls https://dl.k8s.io/release/stable.txt)"

curl -LO "https://dl.k8s.io/release/${KUBECTL_VERSION}/bin/linux/amd64/kubectl"
curl -LO "https://dl.k8s.io/release/${KUBECTL_VERSION}/bin/linux/amd64/kubectl.sha256"
echo "$(cat kubectl.sha256)  kubectl" | sha256sum --check

sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl
rm kubectl kubectl.sha256
```

`stable.txt` sempre aponta para a versão estável mais recente; para fixar uma versão específica (recomendado para reprodutibilidade), substitua a primeira linha por `KUBECTL_VERSION="v1.36.1"` ou a versão desejada, compatível com a versão do K3s em uso (veja a [política de version skew](https://kubernetes.io/releases/version-skew-policy/)).

## Helm

Helm é um gerenciador de pacotes para Kubernetes. Um **chart** reúne templates de manifests, valores padrão e metadados; um arquivo `values.yaml` ou opções `--set` personalizam esses templates; uma **release** é uma instalação identificada desse chart em um cluster. `helm upgrade --install` cria a release quando ela não existe ou gera uma nova revisão quando existe, permitindo consultar o histórico e realizar rollback.

Helm renderiza manifests e os envia à API Kubernetes usando o kubeconfig, mas não mantém um controller próprio reconciliando continuamente a release. Depois da instalação, uma alteração manual no cluster não é corrigida por Helm até uma nova operação; essa é uma diferença importante em relação ao Argo CD. Referência: [introdução ao Helm](https://helm.sh/docs/intro/introduction/).

**Modelo de acesso e privilégios:** mesmo modelo do `kubectl` (RBAC via kubeconfig); um chart mal revisado pode criar qualquer recurso que a identidade ativa tenha permissão de criar, incluindo RBAC mais permissivo para si mesmo. **Riscos:** instalar um chart de um repositório não confiável equivale a rodar manifests arbitrários com os privilégios da identidade ativa; revise o `values.yaml` e, quando possível, o conteúdo do chart antes de instalar. **Licença e plataformas:** Apache 2.0. Linux, macOS e Windows.

Desde o lançamento do Helm v4, o v3 passou a receber apenas manutenção por um período limitado; novas instalações devem usar o v4. Confirme o estado atual de suporte do v3 na [política de releases do Helm](https://helm.sh/docs/topics/release_policy/) antes de optar por instalar a série anterior. A instalação oficial usa o script mantido pelo próprio projeto Helm:

> **Executar em:** máquina onde a CLI será instalada.

```bash
curl -fsSL -o get_helm.sh https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-4
chmod +x get_helm.sh

DESIRED_VERSION=<versão-do-helm> ./get_helm.sh
rm get_helm.sh
```

Fixe `DESIRED_VERSION` com uma versão específica das [releases do Helm](https://github.com/helm/helm/releases) em vez de omiti-la; sem essa variável, o script instala a versão mais recente disponível no momento da execução, o que torna a instalação não reprodutível entre máquinas. Até a escrita deste texto, a versão estável mais recente é `v4.2.3`; confirme se há uma versão mais nova nas releases e leia o changelog antes de fixá-la.

## Argo CD CLI

`argocd` é o cliente da API do Argo CD. Ele permite autenticar no servidor Argo CD, cadastrar repositórios, consultar diferenças e controlar sincronizações. Não substitui `kubectl`: a CLI `argocd` opera os conceitos do Argo CD, enquanto `kubectl` opera qualquer recurso exposto pela API Kubernetes, inclusive os CRDs `Application`.

**Modelo de acesso e privilégios:** autentica contra o servidor Argo CD (usuário local, SSO ou token de API), com autorização controlada pelo RBAC próprio do Argo CD, separado do RBAC do Kubernetes que `kubectl` usa. **Riscos:** um token de API do Argo CD com escopo amplo, vazado, dá controle sobre a sincronização de todas as `Application`s que esse escopo alcança; trate-o com o mesmo cuidado dado a uma credencial de deploy. **Licença e plataformas:** Apache 2.0. Linux, macOS e Windows.

A instalação oficial baixa o binário publicado nas releases do projeto no GitHub:

> **Executar em:** máquina onde a CLI será instalada.

```bash
ARGOCD_VERSION="<versão-do-argocd>"
ARGOCD_ARCH="amd64"  # ou arm64, conforme a arquitetura da máquina

curl -sSL -o argocd \
  "https://github.com/argoproj/argo-cd/releases/download/${ARGOCD_VERSION}/argocd-linux-${ARGOCD_ARCH}"

sudo install -o root -g root -m 0755 argocd /usr/local/bin/argocd
rm argocd
```

Fixe `ARGOCD_VERSION` com uma versão específica das [releases do Argo CD](https://github.com/argoproj/argo-cd/releases); prefira alinhá-la à versão do chart Argo CD já instalado no cluster (veja [instalar o Argo CD](../../../../guides/tasks/gitops/install-argocd/)), já que a CLI e o servidor têm compatibilidade mais estreita entre versões próximas. Até a escrita deste texto, a versão estável mais recente é `v3.4.5`; confirme se há uma versão mais nova nas releases e leia o changelog antes de fixá-la.

## cmctl

`cmctl` é o cliente administrativo do cert-manager. Ele verifica a disponibilidade da API e do webhook, reúne o estado dos recursos envolvidos na emissão de um certificado e permite solicitar uma reemissão manual controlada. A ferramenta usa o mesmo kubeconfig e contexto de `kubectl`; portanto, suas operações ficam limitadas pelas permissões da identidade ativa.

A partir da versão 2, `cmctl` possui ciclo de releases independente do cert-manager. Não presuma que as duas versões precisam ser iguais. A versão de referência deste guia é `v2.5.0`; instale-a por um dos [métodos oficiais](https://cert-manager.io/docs/reference/cmctl/#installation), fixe a versão quando usar o binário de release e valide o arquivo com o `checksums.txt` publicado na mesma release.

Não use a antiga imagem `cert-manager-ctl` como alternativa atual em Docker: o cert-manager 1.14 foi a última versão que a publicou. Este guia usa o binário standalone `cmctl`; o upstream também documenta o [modo de plugin `kubectl cert-manager`](https://cert-manager.io/docs/reference/cmctl/#kubectl-plugin), instalado com o mesmo binário sob o nome esperado pelo `kubectl`.

Os procedimentos de `check api`, diagnóstico e renovação ficam na página do [cert-manager](../../../../guides/tasks/certificates/install-cert-manager/).

**Modelo de acesso e privilégios:** mesmo modelo do `kubectl` (RBAC via kubeconfig); operações de reemissão manual exigem permissão de escrita sobre os recursos `Certificate`/`CertificateRequest` do cert-manager. **Riscos:** nenhum além do já coberto por `kubectl`; é somente uma interface mais direta para operações que já seriam possíveis manipulando os CRDs do cert-manager. **Licença e plataformas:** Apache 2.0. Linux, macOS e Windows.

## longhornctl

`longhornctl` auxilia na preparação e no diagnóstico do Longhorn, principalmente por meio dos testes de preflight usados neste guia. Ele não monta volumes para as aplicações e não substitui a interface CSI; os Pods continuam solicitando armazenamento por PVCs Kubernetes.

**Modelo de acesso e privilégios:** mesmo modelo do `kubectl` para operações no cluster; os testes de preflight também podem exigir acesso direto aos discos do host quando avaliam requisitos de storage local. **Riscos:** nenhum além do já coberto por `kubectl`; os testes de preflight são somente leitura por padrão. **Licença e plataformas:** Apache 2.0. Linux, macOS e Windows.

Depois de instalar as ferramentas, valide as versões:

> **Executar em:** máquina onde as CLIs foram instaladas.

```bash
kubectl version --client
helm version
argocd version --client
cmctl version --client
longhornctl version
```
